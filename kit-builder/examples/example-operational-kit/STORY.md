# Story

A support analyst finishes a messy troubleshooting session.

The issue was solved, but the useful knowledge is scattered across:
- terminal output
- a few copied error messages
- a chat conversation
- memory
- maybe a screenshot

The analyst knows that if this problem happens again, someone will ask:

> What happened last time?

So before moving on, the analyst creates a short troubleshooting note.

They capture:
- what failed
- what was observed
- what was tested
- what fixed it
- what did not fix it
- what should be checked first next time

The note does not need to be perfect.

It needs to be clear enough that a tired person can understand the situation later.

The analyst saves the note in a predictable place so it can be reused during future troubleshooting.
